# Пользовательские сценарии (Use Cases) — АБИС

Кейс 8: Библиотека — выдача и просрочки | Версия 1.0 | Дата: 12.04.2026

|------|----------|-----------------|

## UC-01: Регистрация выдачи книги

**Актор:** Библиотекарь  
**Предусловие:** Читатель и книга существуют в системе  
**Основной поток:**
1. Библиотекарь открывает страницу «Выдачи».
2. Выбирает читателя из списка (поиск по ФИО или reader_id).
3. Выбирает книгу из каталога (поиск по названию или book_code).
4. Система проверяет: экземпляр (book_code) не имеет активной выдачи.
5. Система проверяет: issue_date <= due_date.
6. Библиотекарь подтверждает выдачу.
7. Система создаёт запись в loans: loan_id, reader_id, book_code, issue_date, due_date, return_date=NULL.

**Альтернативный поток (4а):** Если экземпляр уже выдан — система выводит ошибку «Экземпляр имеет активную выдачу».  
**Постусловие:** Новая запись о выдаче создана, книга помечена как выданная.

## UC-02: Оформление возврата книги

**Актор:** Библиотекарь  
**Предусловие:** Существует активная выдача (return_date = NULL)  
**Основной поток:**
1. Библиотекарь находит активную выдачу по reader_id или book_code.
2. Нажимает «Оформить возврат».
3. Система фиксирует return_date = текущая дата.
4. Если return_date > due_date — система рассчитывает days_overdue = return_date - due_date.
5. Запись обновляется в loans.

**Постусловие:** Выдача закрыта, экземпляр снова доступен для выдачи.

---

## UC-03: Формирование отчёта «Просрочки»

**Актор:** Библиотекарь / Администратор  
**Предусловие:** В системе есть записи о выдачах  
**Основной поток:**
1. Пользователь открывает страницу «Просрочки».
2. Задаёт фильтры: период, группа читателя, автор/УДК, интервал просрочки.
3. Система формирует отчёт с полями: reader_id, fio, book_code, title, days_overdue.
4. Отображаются показатели: доля просроченных выдач (%), средняя/медианная просрочка (дни), ТОП-N читателей по просрочке.
5. Пользователь экспортирует отчёт в CSV.

**Постусловие:** Отчёт сформирован, CSV-файл доступен для скачивания.

## UC-04: Импорт данных из CSV

**Актор:** Администратор  
**Предусловие:** CSV-файлы подготовлены по шаблону (books.csv, readers.csv, loans.csv)  
**Основной поток:**
1. Администратор открывает страницу «Интеграция».
2. Загружает CSV-файл (books/readers/loans).
3. Система валидирует данные: обязательные поля, домены, ссылочная целостность.
4. Идемпотентный импорт: при повторной загрузке дубли не создаются (проверка по PK).
5. Система отображает результат: количество добавленных/пропущенных/ошибочных записей.

**Альтернативный поток (3а):** Ошибки валидации — система выводит список ошибок с номерами строк.  
**Постусловие:** Данные загружены в систему без дублей.

---

## UC-05: Просмотр каталога читателем

**Актор:** Читатель  
**Предусловие:** Читатель авторизован в системе  
**Основной поток:**
1. Читатель открывает каталог.
2. Фильтрует книги по автору, году, наличию.
3. Просматривает карточку книги (title, author, year, UDK, статус наличия).
4. В профиле видит историю своих выдач с датами и статусами.

**Постусловие:** Читатель получил актуальную информацию о фонде.
