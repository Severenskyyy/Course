## 4. Потоки данных между модулями

| Поток                              | Инициатор         | Протокол        | Формат          |
|------------------------------------|-------------------|-----------------|-----------------|
| UI → API (обычные операции)         | Frontend          | HTTPS           | JSON (REST)     |
| API → БД                            | Backend           | In-process      | SQL             |
| 1С → API (планировщик, опционально) | 1С                | HTTPS           | JSON            |
| 1С ↔ CSV ↔ Backend                  | Администратор     | File            | CSV (UTF-8, `;`) |
| Backend → 1С (отчёт)                | Библиотекарь      | File export     | CSV             |

## 5. Архитектурные паттерны

1. **Layered architecture** (UI / API / Service / Repo / DB).
2. **DI через Depends** (FastAPI) — сессии, текущий пользователь.
3. **Repository pattern** в бэкенде — запросы в отдельных функциях, не в роутерах.
4. **DTO (Pydantic) vs Model (SQLAlchemy)** — чёткое разделение представлений.
5. **Idempotent import** — натуральные ключи в качестве PK CSV-контрактов.

## 6. Решения об отклонённых альтернативах (ADR-краткий)

| Вариант рассмотренный      | Решение              | Причина                                       |
|-----------------------------|----------------------|-----------------------------------------------|
| PostgreSQL vs SQLite        | SQLite               | Учебный стенд, нулевой порог входа            |
| Django REST vs FastAPI      | FastAPI              | Легче, автогенерация OpenAPI                  |
| Бронирования в MVP          | Отложено в Could     | Увеличивает объём в 1.5×, не обязательно по ТЗ |
| Prometheus/Grafana          | Отказ                | Вне скоупа учебного проекта                   |

## 7. Вывод

Архитектура удовлетворяет всем функциональным требованиям Кейса 8, НФТ (производительность, надёжность, трассируемость, разграничение ролей), а модульная структура позволяет параллельно дорабатывать компоненты без конфликтов.
