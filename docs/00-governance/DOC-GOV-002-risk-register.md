---

## 1. Метод оценки

Используется метод **Харрингтона**: каждый риск оценивается двумя параметрами по шкале 1–5:

- **Pr (probability)** — вероятность наступления: 1 = почти невозможно, 5 = почти наверняка.
- **Im (impact)** — влияние на проект: 1 = незначительное, 5 = критическое.

Балл риска: **R = Pr × Im** (от 1 до 25).

| Зона | Балл | Цвет | Действие |
|---|---|---|---|
| Низкий | 1–6 | 🟢 | Принять, мониторить |
| Средний | 7–12 | 🟡 | Снижать или передавать |
| Высокий | 13–25 | 🔴 | Активные меры, эскалация куратору |

## 3. Детальные карточки активных рисков

### Р-02. Расхождение схем 1С и API после правок (КРИТИЧЕСКИЙ)

- **Описание.** При правках в 1С (новое поле в справочнике или документе) backend не обновляется, импорт CSV ломается.
- **Триггеры.** Каждое изменение метаданных в 1С.
- **Меры.**
  - Контракты CSV зафиксированы в `DOC-INT-002` и в `1c/ExchangeFormats/*.csv`.
  - Любая правка схемы → одновременный PR в 1С + backend + DOC-INT-002.
  - Регрессия `pytest -v` после каждого изменения.
- **Закрытие.** Не закрывается до конца проекта; постоянный мониторинг.

### Р-01. Уход / болезнь участника команды

- **Описание.** Из-за сессии или болезни один участник не может выполнять задачи 1+ неделю.
- **Меры.** Парные зоны RACI; документация в репозитории, не «в голове»; тимлид имеет доступ ко всем веткам.
- **План B.** Перераспределение задач за 24 часа; сдвиг внутренних дедлайнов на ≤ 1 неделю.

### Р-03. Несовпадение кодировки CSV

- **Меры.** UTF-8 BOM по умолчанию; cp1251 — по явному флагу `Charset` в заголовке; тест `test_csv_charset_detection`.

### Р-04. Сбой 1С на проекторе

- **Меры.** Запасной экспорт CSV в `defense/h4/snapshot/`; видео-демо 1С (5 минут); ноутбук тимлида с проверенной 1С.

### Р-05. Срыв подготовки к H4

- **Меры.** Заморозка скоупа фич с 19.04.2026 (после H3); только багфиксы; релиз-кандидат `v2.2-lite-rc` за неделю до H4.

### Р-06. Падение API во время демо

- **Меры.** Запасной локальный билд; прогон `pytest -v` (17/17) перед демо; snapshot БД после T1-загрузки.

### Р-08. Изменение требований куратором

- **Меры.** MoSCoW зафиксирован после H1 (DOC-REQ-004); change-requests через issue, обсуждение ≤ 24 часа.

### Р-10. Несовпадение зачётных дат

- **Меры.** Раннее согласование графика защит с куратором, резервный исполнитель в команде.
