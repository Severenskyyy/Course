## 1. Цель документа

Зафиксировать уроки и решения по работе команды за все 8 недель проекта. Документ готовится к итоговой защите (Хакатон 4) и служит основой для улучшения процессов в следующих семестрах.

## 2. Формат ретроспективы

Использован формат **Start / Stop / Continue** в связке с короткими «что сработало / что не сработало» после каждого хакатона.

## 4. Start / Stop / Continue

### Начать (Start)
- Писать тест на API **до** реализации эндпоинта (TDD-lite) — это дало «зелёную ленту» на 4-й неделе.
- Регулярная сверка BPMN с экранами фронта раз в неделю.
- Использовать issues-доску как единый backlog (задачи ОПД и хакатоны).

### Прекратить (Stop)
- Не пытаться сделать «красивый» BPMN сразу — сначала черновой, потом детализация.
- Не хранить рабочие документы в чате — сразу в репозиторий.
- Перестать редактировать компоненты React через str-replace правки (Теx-лид переписал целиком — быстрее).

### Продолжить (Continue)
- Практика ежедневных 10-минутных стендапов в Telegram-чате.
- Подробные PR-описания (что, зачем, как тестировать).
- Ответственность «один артефакт — один владелец».

## 5. Lessons learned

### Технические
1. **Единый формат CSV с самого начала** — все расхождения между 1С и бэкендом убрали за счёт единого `ExchangeFormats/` и шаблонов.
2. **Idempotent import** дался не сразу — потребовал ключа «loan_id» как натурального ID.
3. **OpenAPI → контракт для фронта** сэкономил 2–3 дня коммуникации.
4. **Generate synthetic data + golden** — лучшее, что можно сделать на старте, чтобы бэкенд и отчёты не «плыли».

### Процессные
1. Role redistribution после 3-й недели (Коптилин ↔ Гуков) повысил скорость, хотя временно просел фронт.
2. Заморозка MoSCoW после ХК-1 защитила от «раздувания» скоупа.
3. Контрольные точки хакатонов = маленькие MVP, это позволило не сваливаться в большой риск в конце.

### Личное
- Параллельная сессия (у 2-х членов команды были экзамены) требует буфера в планах.
- Учебная 1С-лицензия вовремя оказалась ограниченной — следующий семестр заранее согласовывать с кафедрой.
