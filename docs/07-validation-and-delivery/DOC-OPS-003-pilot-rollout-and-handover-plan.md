# План пилотного развёртывания и передачи — АБИС (Кейс 8)

| Версия | Статус | Дата |
|---|---|---|
| v1.0 | Final | 2026-05-09 |

**Авторы документа.** Кирбитов А.Е. — план и зоны ответственности; Гуков Я.Р. — backend-операции; Коптилин И.Р. — frontend; Рабаданов А.С. — 1С-сторона.

**Назначение.** Описать, как АБИС переходит из учебного MVP в режим пилотной эксплуатации в одной учебной библиотеке, и как происходит передача системы заказчику. Документ отделён от технического runbook (DOC-OPS-002), который сфокусирован на *как развернуть*; здесь — *кто, когда и в каком порядке начинает пользоваться*.

---

## 1. Цели пилота

- Убедиться, что работающая модель «выдача → возврат → просрочка» закрывает реальные операции библиотекаря.
- Замерить KPI на живых данных в течение 4 недель.
- Получить обратную связь и список улучшений для версии 3.0.
- Передать систему и документацию заказчику, провести обучение.

Пилот не выходит за рамки одной учебной библиотеки и одного компьютера библиотекаря.

---

## 2. Состав пилота

| Параметр | Значение |
|---|---|
| Целевая аудитория | 1 библиотекарь (Анна) + 1 администратор (Дмитрий) |
| Локация | Учебная библиотека (1 рабочее место) |
| Объём данных | До 500 книг, до 200 читателей, до 1000 выдач/мес |
| Длительность | 4 недели (после защиты H4) |
| Поддержка | Команда проекта в режиме on-call (≤ 4 ч на ответ) |

---

## 3. Этапы внедрения

### Этап 0. Подготовка (за 1 неделю до старта)

- [ ] Заказчик готовит выгрузку текущего каталога из Excel в формате CSV.
- [ ] Команда настраивает `production`-конфигурацию (см. DOC-OPS-002).
- [ ] Заводятся учётки `admin`/`librarian`/`reader` с временными паролями и инструкцией по смене.
- [ ] Бэкап-стратегия: ежедневный snapshot БД через `tools/backup.py` (cron на машине заказчика).

### Этап 1. Загрузка данных (день 1)

1. Импорт `books.csv` через IntegrationPage → проверка количеств.
2. Импорт `readers.csv` → проверка количеств.
3. Загрузка исторических `loans.csv` (открытые на момент старта) — опционально, по решению библиотекаря.
4. Сверка с Excel-журналом: 5–10 контрольных позиций.

### Этап 2. Параллельная работа (недели 1–2)

- Библиотекарь ведёт **обе** системы (Excel + АБИС) на одних и тех же выдачах.
- Раз в день — сверка количеств активных выдач.
- При расхождениях — разбор и фикс в АБИС.

Цель этапа — научиться, не потеряв надёжность Excel.

### Этап 3. Полный переход (недели 3–4)

- Excel переводится в режим «справочный архив».
- Все новые выдачи/возвраты только через АБИС.
- Дашборд KPI стал основным источником цифр для отчётов.
- Раз в неделю — экспорт CSV в 1С (контур I-04, I-05).

### Этап 4. Подведение итогов (после 4 недели)

- Снять метрики: KPI-1 (доля просрочек), KPI-2 (скорость отчёта), время на выдачу одной книги.
- Собрать обратную связь от библиотекаря (чек-лист в Приложении A).
- Зафиксировать список улучшений → бэклог версии 3.0.
- Решение «пилот успешен / нужны доработки / откат».

---

## 4. Критерии успеха пилота

| Критерий | Целевое значение |
|---|---|
| Время на оформление одной выдачи | ≤ 30 секунд |
| Доля просроченных выдач (KPI-1) | ≤ 15% |
| Расхождения между АБИС и 1С после еженедельного обмена | 0 записей |
| Удовлетворённость библиотекаря (1–5) | ≥ 4 |
| Количество критических инцидентов за пилот | 0 |
| Количество замечаний из обратной связи | ≤ 10 (всё non-critical) |

---

## 5. Риски и митигация (пилот)

| Риск | Митигация |
|---|---|
| Потеря данных | Ежедневный snapshot БД + ручной бэкап раз в неделю на внешний носитель |
| Отказ от системы из-за неудобств | Параллельная работа на 1–2 неделях, ежедневная связь с командой |
| Перегрузка из-за неподготовленных данных | Этап 0 с предварительной выгрузкой и валидацией |
| Сбой 1С-обмена в первый месяц | Команда лично присутствует на первом цикле обмена; CSV формируются вручную |

## 6. Передача системы заказчику (handover)

После успешного пилота:

| Артефакт | Что передаётся | Получатель |
|---|---|---|
| Исходный код | Git-репозиторий, релизный тэг `v2.2-lite-final` | Заказчик (read-only fork) |
| Сборка | `dist/index.html`, `backend/app/`, `1c/1Cv8.dt` | Администратор |
| Документация | Папка `docs/` (33 MD + 3 DOCX) | Заказчик |
| Тестовые наборы | `data/T1`, `data/golden`, `tests/validators/` | Администратор |
| Инструкции | `1c/README.md`, `backend/README.md`, корневой `README.md` | Все |
| Учётки | Список с временными паролями (на бумаге, в конверте) | Администратор |
| Контактные данные команды | E-mail, окно on-call (1 месяц) | Заказчик |

Подписание акта передачи — на финальной встрече, отдельным документом.

## 7. План обучения (training)

| Сессия | Целевая роль | Длительность | Темы |
|---|---|---|---|
| T-1 | Администратор | 90 мин | Установка, бэкап, 1С-обмен, журнал, смена паролей |
| T-2 | Библиотекарь | 60 мин | Каталог, читатели, выдачи, возвраты, отчёт «Просрочки» |
| T-3 | Библиотекарь | 30 мин (повтор) | Сложные кейсы: BR-01..BR-06, как искать, как фильтровать |
| T-4 | Заказчик (директор) | 30 мин | Дашборд KPI, отчёты, экспорт |

Материалы тренинга: одностраничный «cheat sheet» (PDF) + видео-демо (5 мин, см. `docs/90-checkpoints/`).

---

## 8. Поддержка после пилота

- **Месяц 1 после передачи:** on-call команды (e-mail, ответ ≤ 24 ч).
- **Месяц 2–3:** только критические инциденты, ответ ≤ 72 ч.
- **После 3 месяцев:** проект считается переданным; дальнейшие доработки — отдельным договором.

## Приложение A. Чек-лист обратной связи (4 неделя пилота)

1. Стало ли проще оформлять выдачи по сравнению с Excel? (1–5)
2. Сколько раз в неделю вы пользуетесь дашбордом KPI? (число)
3. Случалось ли, что система мешала работе? (Да/Нет; описание)
4. Доверяете ли вы цифрам в АБИС больше, чем Excel? (Да/Нет/Частично)
5. Какие функции хотелось бы добавить в первую очередь? (свободный ответ)
6. Готовы ли продолжить пользоваться системой ещё 3 месяца? (Да/Нет)
