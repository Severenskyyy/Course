# Маппинг полей 1С ↔ Backend API — АБИС (Кейс 8)

| Версия | Статус | Дата       |
|--------|--------|------------|
| v1.0   | Final  | 2026-04-27 |

## 1. Назначение

Документ — единый источник правды по соответствию полей между объектами 1С:Предприятия и REST-API бэкенда FastAPI. Используется 1С-разработчиком, backend-разработчиком и интегратором.

## 2. Принципы

1. **Язык хранения:** внутри 1С — русский (Наименование, Автор), в API и БД — английский (title, author).
2. **Преобразование дат:** все даты по обеим сторонам хранятся без времени (`YYYY-MM-DD`). В 1С — тип «Дата» с фракцией `Date`.
3. **Кодировка:** UTF-8 в обе стороны.
4. **Ключи:** натуральные (`book_code`, `reader_id`, `loan_id`) выступают как идемпотентные идентификаторы.

## 5. Маппинг `loans` (комплексный)

`loans` в API — единая сущность, в 1С — связка **Документ.ВыдачаКниги** + **Документ.ВозвратКниги** + **Регистр.Выдачи**.

| API           | Backend           | 1С                                                          | Примечание                                   |
|---------------|-------------------|-------------------------------------------------------------|----------------------------------------------|
| `loan_id`     | `loans.loan_id`   | Документ.ВыдачаКниги.Номер                                  | PK ↔ уникальный номер документа              |
| `reader_id`   | `loans.reader_id` | Документ.ВыдачаКниги.Читатель.Код                           | FK                                           |
| `book_code`   | `loans.book_code` | Документ.ВыдачаКниги.Книга.Код                              | FK                                           |
| `issue_date`  | `loans.issue_date`| Документ.ВыдачаКниги.Дата                                   | ISO ↔ Дата                                   |
| `due_date`    | `loans.due_date`  | Документ.ВыдачаКниги.ПлановаяДатаВозврата                   | ISO ↔ Дата                                   |
| `return_date` | `loans.return_date`| Документ.ВозвратКниги.ДатаВозврата (если возврат есть)     | NULL ↔ отсутствие документа возврата         |
| `status`      | (вычисляется)     | (вычисляется)                                               | `active / returned / overdue`                |
| `days_overdue`| (вычисляется)     | Документ.ВозвратКниги.ДнейПросрочки                         | На стороне BE рассчитывается в SQL-запросе   |
| `created_by`  | `loans.created_by`| Документ.ВыдачаКниги.Библиотекарь                           | user_id ↔ ссылка на пользователя 1С          |

### 5.1 Логика синхронизации возврата

```
BE состояние loans           1С состояние
────────────────────         ─────────────────────────────────────
return_date = NULL    ◄────► Только Документ.ВыдачаКниги проведён
return_date = <дата>  ◄────► ВыдачаКниги + ВозвратКниги проведены
```

## 6. Вычисляемые поля (не маппятся напрямую)

| Поле          | Формула BE                                     | Формула 1С                                                 |
|---------------|------------------------------------------------|------------------------------------------------------------|
| `status`      | `CASE` WHEN return_date NOT NULL THEN 'returned' WHEN today > due_date THEN 'overdue' ELSE 'active' | Аналогично — в запросе СКД                       |
| `days_overdue`| `GREATEST(0, COALESCE(return_date, :today) - due_date)` | `ОбщегоНазначения.РассчитатьДниПросрочки()`    |

## 7. Проверки целостности при обмене

Перед записью в 1С (из импорта CSV или API):
1. Книга с `book_code` существует → если нет, создать справочник с заполнением из BE.
2. Читатель с `reader_id` существует → если нет, отказ (BR-03).
3. Идемпотентность: если документ с тем же `loan_id` уже существует — UPSERT.
4. Связка возврата с выдачей проверяется через реквизит `ДокументВыдачи`.

## 8. Конфликты и стратегии разрешения

| Конфликт                         | Стратегия                                                |
|----------------------------------|----------------------------------------------------------|
| Один `book_code`, разные `title` | Выигрывает последняя загрузка (Last-Write-Wins), фикс в `audit_log` |
| В 1С есть, в BE нет (и наоборот) | Система источник истины — BE; при расхождении — запись в `audit_log` |
| Расхождение дат                  | Фиксируется как ошибка импорта, строка отклоняется       |

## 9. Примечания для разработчиков

- При изменении схемы любого объекта из 1С или BE — **обновить этот документ в том же PR**.
- 1С-разработчик поддерживает `ExchangeFormats/*.csv`, backend-разработчик поддерживает `spec/openapi.yaml`. Любое расхождение между ними — bug высокого приоритета.
