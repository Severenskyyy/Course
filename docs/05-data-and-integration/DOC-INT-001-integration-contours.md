# Контуры интеграции и сценарии обмена — АБИС (Кейс 8)

| Версия | Статус | Дата       |
|--------|--------|------------|
| v1.0   | Final  | 2026-04-25 |

## 1. Назначение

Документ описывает два основных контура интеграции в системе АБИС: внутренний (Frontend ↔ Backend) и внешний (Backend ↔ 1С:Предприятие через CSV/JSON). Также описаны сценарии обмена в рамках типовой работы библиотекаря и администратора.

## 2. Контур 1: Frontend ↔ Backend

### 2.1 Архитектура

```
┌─────────────┐      HTTPS + JSON (REST)        ┌─────────────┐
│   React     │◄──────────────────────────────►│   FastAPI   │
│   (Vite)    │                                 │ (Uvicorn)   │
└─────────────┘                                 └─────────────┘
      ▲                                                │
      │  Авторизация: JWT (Bearer)                     │
      │  Формат: application/json                      │
      │  Кодировка: UTF-8                              ▼
      │                                          ┌─────────┐
      │                                          │ SQLite  │
      │                                          └─────────┘
```

### 2.2 Спецификация — OpenAPI

Полная спецификация хранится в `spec/openapi.yaml`. Основные группы эндпоинтов:

| Группа            | Базовый путь               | Назначение                                  |
|-------------------|----------------------------|---------------------------------------------|
| Auth              | `/api/auth/*`              | Логин, refresh-токен, logout                |
| Books             | `/api/books/*`             | CRUD по книгам                              |
| Readers           | `/api/readers/*`           | CRUD по читателям                           |
| Loans             | `/api/loans/*`             | Выдачи, возвраты, история                   |
| Reports           | `/api/reports/*`           | Отчёт «Просрочки», KPI, экспорт             |
| Import/Export     | `/api/import-export/*`     | Импорт CSV, экспорт CSV                     |

### 2.3 Роли и права доступа

- **admin** — полный доступ ко всем эндпоинтам.
- **librarian** — CRUD по книгам/читателям, выдачи/возвраты, отчёты (без `DELETE` на `users`).
- **reader** — только просмотр каталога и своих выдач.

## 3. Контур 2: Backend ↔ 1С:Предприятие

### 3.1 Архитектура обмена

```
        ┌───────────────┐   CSV (UTF-8, разделитель ;)   ┌────────────┐
        │   FastAPI     │◄───────────────────────────────│  1С:ERP    │
        │  + SQLite     │───────────────────────────────►│  8.3       │
        └───────────────┘      JSON (опционально)        └────────────┘
              ▲
              │
              │ CSV-файлы (ручной или автомат. режим)
              │
        ┌───────────────┐
        │ Администратор │
        └───────────────┘
```

### 3.2 Режимы обмена

| Режим         | Направление  | Формат | Инициатор     | Частота             |
|---------------|--------------|--------|---------------|---------------------|
| **Импорт каталога** | 1С → Backend | CSV    | Администратор | При актуализации   |
| **Импорт читателей**| 1С → Backend | CSV    | Администратор | Еженедельно        |
| **Импорт выдач**    | 1С → Backend | CSV    | Администратор | Однократно (миграция)|
| **Экспорт просрочек**| Backend → 1С | CSV    | Библиотекарь | По запросу          |
| **Сверка KPI (опц.)**| Backend → 1С | JSON   | Планировщик  | Ежедневно (Could)   |

### 3.3 Идемпотентность

Все импорты **идемпотентны**:
- `books`: ключ `book_code`, UPSERT по совпадению.
- `readers`: ключ `reader_id`, UPSERT.
- `loans`: ключ `loan_id`, при совпадении обновляется только `return_date`.

Проверка идемпотентности — обязательный тест (`tests/test_import_idempotent.py`).

## 4. Типовые сценарии обмена

### Сценарий A: Актуализация каталога

1. 1С-администратор выгружает `books.csv` из типовой конфигурации 1С.
2. Через веб-интерфейс АБИС (`Integration → Import`) загружает файл.
3. Backend валидирует: контроль типов, диапазона годов (`1000 <= year <= 2100`), уникальности `book_code`.
4. Ответ API: `{"created": N, "updated": M, "skipped": K, "errors": [...]}`.
5. В `audit_log` появляется запись с `entity_type='books'`, `action='import'`, `user_id=<admin>`.

### Сценарий B: Экспорт отчёта «Просрочки»

1. Библиотекарь открывает экран `OverduePage`.
2. Применяет фильтры (группа, период, UDK).
3. Нажимает «Экспорт CSV» → вызов `GET /api/reports/overdue/export?...`.
4. Backend формирует CSV с полями `reader_id;fio;group;book_code;title;issue_date;due_date;days_overdue`.
5. Ответ: `Content-Type: text/csv`, `Content-Disposition: attachment; filename=overdue_YYYYMMDD.csv`.
6. 1С-подсистема принимает тот же файл (`ЭкспортПросрочек.ЭкспортироватьВCSV`).

## 5. Безопасность обмена

| Контур         | Меры                                                                           |
|----------------|--------------------------------------------------------------------------------|
| Frontend ↔ Backend | JWT (алг. HS256), expiration 15 минут, refresh 7 дней, HTTPS (в проде)      |
| Backend ↔ 1С   | Обмен по локальной сети; в MVP без шифрования, в проде — VPN/mTLS               |
| CSV-файлы      | UTF-8 BOM обязателен для совместимости Excel; значения с `;` экранируются       |

## 6. Мониторинг обмена

- Все операции импорта/экспорта логируются в `audit_log`.
- Ошибочные строки CSV попадают в «журнал проблемных строк» и возвращаются в ответе API.
- Метрики объёма обмена доступны на дашборде KPI (опционально).
